# Автообновление десктоп-лаунчера

Дата: 2026-06-11. Статус: утверждено.

## Цель

Лаунчер (launcher-slint) проверяет обновления при запуске и во время работы,
скачивает новую версию в фоне и предлагает перезапуск. Релизы заливаются
через дашборд. Поддерживаются обязательные (форс) обновления — рычаг для
античита: старые версии не получают launch-token и не могут запускать игру.

Платформы: linux-x64 и windows-x64. Механизм: самозамена бинарника
(скачал → сверил SHA-256 → подменил себя → перезапуск по кнопке).
Безопасность v1: SHA-256 + HTTPS, без криптоподписи (можно добавить позже).

## 1. Бэкенд: модель и хранилище

Новый доменный пакет `internal/launcherrelease` (паттерн `service.go` + `handler.go`).
Модели в `internal/models`, схема через GORM AutoMigrate:

- `LauncherRelease`: `ID` (uuid), `Version` (semver-строка, уникальная),
  `Changelog` (text), `Mandatory` (bool), `IsActive` (bool, default true),
  `CreatedAt`.
- `LauncherReleaseFile`: `ID`, `ReleaseID` (FK), `Platform`
  (`linux-x64` | `windows-x64`), `FileName`, `Size`, `SHA256`.

Бинарники на диске: `backend/storage/releases/<version>/<platform>/<filename>`.
Каталог в git не входит, `deploy.sh` его не затирает (как `storage/profiles`).

## 2. Бэкенд: API

Публичные эндпоинты (без JWT — обновление должно работать до логина):

- `GET /api/launcher/update?platform=<p>&version=<v>` →
  `{update_available, latest_version, mandatory, changelog, download_url, sha256, size}`.
  `latest_version` — максимальная активная версия, у которой есть файл под
  запрошенную платформу. `mandatory` = true, если в интервале
  (версия клиента, latest] есть хоть один активный релиз с флагом `Mandatory`.
  Сравнение версий — корректный semver-компаратор (не строковое сравнение).
- `GET /api/launcher/download/:version/:platform` — отдача бинарника через
  `fiber.SendFile` с защитой пути (как в profiles.Download).

Админские (под существующей admin-авторизацией, как `/api/admin/profiles`):

- `GET /api/admin/releases` — список с файлами.
- `POST /api/admin/releases` — multipart: поля `version`, `changelog`,
  `mandatory` + файлы по платформам (минимум один). SHA-256 и размер считает
  бэкенд при приёме. Валидация: уникальность версии, формат semver,
  допустимые платформы, лимит размера (200 МБ).
- `PATCH /api/admin/releases/:id` — переключение `mandatory` / `is_active`.
- `DELETE /api/admin/releases/:id` — удаление записи и файлов с диска.

События: при создании/активации релиза бэкенд публикует в существующий
SSE-брокер (`events.Broker`, канал `/api/profiles/events`) событие с типом
`launcher-release`. Лаунчер различает тип события по payload (сейчас он на
любое событие перезапрашивает профили — это сохраняется для событий профилей).

## 3. Серверный рычаг форс-апдейта

Клиентская блокировка по флагу `mandatory` — вежливость; старый бинарник
может её игнорировать. Настоящий enforcement — серверный:

- Лаунчер шлёт заголовок `X-Launcher-Version` во всех запросах к API.
- Эндпоинт выдачи launch-token античита отвечает `426 Upgrade Required`,
  если версия клиента ниже минимальной обязательной (вычисляется по активным
  mandatory-релизам). Без launch-token игровая сессия не верифицируется —
  тот же рычаг, что у античита.
- Запросы без заголовка (лаунчеры ≤ 0.1.0) считаются версией `0.0.0` и
  блокируются, как только появляется первый обязательный релиз.

## 4. Лаунчер (Rust, launcher-slint)

Версия в рантайме: `env!("CARGO_PKG_VERSION")`. Новый модуль `src/updater.rs`.

Триггеры проверки:
- при старте, до логина (эндпоинт публичный);
- по SSE-событию `launcher-release` из существующего слушателя;
- фоновый опрос раз в 30 минут как страховка (паттерн `thread::spawn`,
  как у SSE-слушателя, с тем же механизмом поколений для shutdown).

Поток обновления:
1. `GET /api/launcher/update` с платформой (определяется при компиляции:
   `cfg!(target_os)`) и текущей версией.
2. Если есть обновление — скачивание существующим `download_client()` во
   временный файл рядом с бинарником (`<exe>.update.partial`), сверка SHA-256.
3. Подмена:
   - Linux: выставить права на исполнение, атомарный `rename` поверх
     текущего исполняемого файла.
   - Windows: переименовать работающий `launcher.exe` → `launcher.exe.old`
     (Windows разрешает rename запущенного exe), новый файл — на его место.
4. UI-баннер: «Скачивается обновление X.Y.Z…» → «Обновление готово —
   Перезапустить». По кнопке: `Command::new(текущий_путь).spawn()` + выход.
5. При каждом старте: удалить остатки `*.old` и `*.update.partial`
   (ошибки удаления игнорируются — файл мог держать завершающийся процесс).

Mandatory-обновление: кнопка запуска игры в UI блокируется до перезапуска
(плюс серверный отказ в launch-token из §3).

Fallback: если каталог бинарника недоступен на запись — самозамена
невозможна; баннер со ссылкой на ручное скачивание.

## 5. Dashboard

Новая страница «Релизы» (в существующем layout админки):
- таблица: версия, дата, платформы, флаги «обязательный»/«активен», changelog;
  действия — переключение флагов, удаление (с подтверждением через
  существующие модалки);
- форма создания: версия, changelog, чекбокс «обязательный», два file-input
  (linux-x64, windows-x64; минимум один файл);
- загрузка через multipart с индикатором прогресса. Текущий `api()`-хелпер
  шлёт только JSON — рядом добавляется хелпер для multipart (XHR или fetch
  с ReadableStream для прогресса).

## 6. Ошибки и крайние случаи

- Несовпадение SHA-256 / обрыв скачивания → удалить `.partial`, показать
  ошибку в баннере, повторить при следующем триггере.
- Дубль версии или не-semver при заливке → 400 с понятным сообщением.
- Деактивация релиза (`is_active=false`) → `/update` его больше не отдаёт
  (откат публикации).
- Скачанный, но не применённый апдейт при выходе из лаунчера — `.partial`
  подчищается при следующем старте, обновление скачается заново.

## 7. Тесты

- Go: юнит-тесты сервиса (semver-сравнение, вычисление mandatory-флага,
  создание/валидация релиза, выбор latest по платформе) и хендлеров.
  Запуск через Docker `golang:1.26-bookworm` (локального go нет).
- Rust: юнит-тесты сравнения версий и вспомогательной логики updater.
- Dashboard: `npm run build` (тайпчек).

## Вне скоупа v1

- macOS / linux-arm64 (структура `Platform` это уже допускает).
- Криптоподпись релизов (ed25519) — место для неё предусмотрено
  (добавить поле `Signature` в `LauncherReleaseFile`).
- Каналы релизов (beta/stable).
- Автозаливка из `build-player-launcher.sh` по API-токену.
