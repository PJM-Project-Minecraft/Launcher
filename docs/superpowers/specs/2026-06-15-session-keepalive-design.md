# Фикс «Недействительная сессия» — keepalive от лаунчера + неубиваемый агент

Дата: 2026-06-15
Ветка: `fix/session-keepalive`

## Проблема (root cause, подтверждён логами)

Игровая yggdrasil-сессия живёт со скользящим TTL (15 мин) и продлевается
heartbeat'ом Java-агента. Но **heartbeat-тред агента стабильно умирает через
~90 с (3 пинга) тихо** — без stack trace в stderr, т.е. через `InterruptedException`
(единственный молчаливый выход из `while(true)` в `Agent.startHeartbeat`). В тяжёлом
модовом окружении (Sinytra Connector мостит Fabric→NeoForge, дёргает класслоадеры/
модули) что-то прерывает наш daemon-тред.

После смерти heartbeat backend-reaper через 90 с молчания **необратимо удаляет
сессию** (`InvalidateByNonce`). Игрок, уже подключённый, играет дальше (сервер не
перепроверяет), но при **любом реконнекте / новом заходе** `join` не находит
сессию → 403 → клиент показывает «Недействительная сессия». Долгая загрузка просто
гарантированно проходит дольше окна жизни сессии.

Соотнесение клиент↔сервер по одному запуску подтвердило: `touched=true` на всех
heartbeat'ах (backend-механика `TouchByNonce` исправна), heartbeat обрывается на
3-м пинге, через 98 с reaper рубит сессию, спустя ~1.5 мин три `join` подряд ловят
«сессия не найдена».

## Решение

Развязать **продление сессии** (надёжный Rust-лаунчер) от **enforcement-живости
агента** (хрупкий JVM-тред) и сделать тред неубиваемым.

### 1. Лаунчер держит сессию живой (ядро)
Пока процесс игры жив, лаунчер каждые **120 с** шлёт keepalive по `nonce`. На выходе
из игры — останавливается, дальше идёт существующий `invalidate`. По `nonce`, а не
`accessToken`, потому что `nonce` переживает `/authserver/refresh` (authlib-injector
может менять токен), а `TouchByNonce` к этому стабилен.

### 2. Backend: keepalive-эндпоинт
`POST /api/yggdrasil/launcher-session/keepalive` под JWT-middleware. Тело `{nonce}` →
`Store.TouchByNonce(nonce)`. `204` если продлил, `404` если сессии нет/истекла
(истёкшую не воскрешаем — kick остаётся окончательным).

### 3. Backend: reaper → мягкий детект
`reapStale` больше **не** зовёт `InvalidateByNonce`. Вместо этого — low-severity
алерт «agent-heartbeat-silent» (дедуп: один раз на nonce) + `slog`. Сессию не трогает.
`InvalidateByNonce` остаётся только в detect-kick (`EvaluateKick`) — реальный чит
по-прежнему мгновенно гасит сессию.

### 4. Агент: неубиваемый heartbeat + самодиагностика
- Тело итерации heartbeat выносится в отдельную тестируемую функцию.
- Цикл оборачивается так, что переживает `Throwable` (continue, не выход) и
  `InterruptedException` (сбросить флаг, continue).
- При перехвате — один раз (дедуп) отправить low-severity diag-детект
  `agent-thread-recovered` с классом+сообщением причины: в логах прода наконец
  станет виден механизм interrupt. Безопасно (не банит).
- То же лечение для `event-poller`-треда (нативные детекты).

## Поток данных (итог)
Старт: `launcher-session(nonce)` → агент `confirm` ⇒ `Verified` → игра.
Игра: лаунчер `keepalive(nonce)` каждые 120 с ⇒ `TouchByNonce` продлевает; агент-
heartbeat дублирует (не критичен для входа). Молчание агента ⇒ алерт, не kill.
Чит ⇒ `InvalidateByNonce` (kill). Выход ⇒ `invalidate`.

## Тестирование
- Backend: keepalive продлевает живую сессию / no-op на истёкшей; reaper алертит и
  **не** удаляет сессию (но detect-kick по-прежнему удаляет).
- Агент: тело heartbeat-итерации устойчиво к брошенному `Throwable` (не выходит).
- Лаунчер: keepalive-цикл пингует пока флаг взведён и останавливается на сбросе.

## Развёртывание
- Backend: обычный деплой (прод вернуть с diag-ветки на `main`+фикс; DIAG-логи уйдут).
- `agent.jar`: пересборка `anticheat-agent/build.sh`, заливка в `backend/data/`,
  игроки получат новый по SHA-манифесту при следующем запуске.
- Лаунчер: новый релиз (сначала необязательный — гоча mandatory из CLAUDE.md).

## Не входит в объём (осознанно)
- Точный механизм interrupt будет подтверждён диагностикой по факту, не до фикса.
- Подход C выбран вместо смягчения только backend — лаунчер как источник живости
  надёжнее JVM-треда.
