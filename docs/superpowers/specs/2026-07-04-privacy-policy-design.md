# Политика конфиденциальности с обязательным согласием — дизайн

Дата: 2026-07-04
Статус: утверждён

## Цель

Добавить Политику конфиденциальности, которую обязан принять каждый игрок — и в
десктоп-лаунчере, и в Telegram-боте — прежде чем пользоваться сервисом. Политика
описывает реальный сбор данных проекта, включая скриншоты экрана во время игровой
сессии (механизм `/api/anticheat/screenshot/*` уже работает в проде). Согласие
фиксируется на сервере с журналом (кто, когда, какую версию, откуда) и форсируется
серверным блоком запуска игры — обойти клиентом нельзя.

## Ключевые решения (зафиксированы с пользователем)

- Каноничный текст — **на бэкенде**, отдаётся по API + публичная HTML-страница.
- Охват — **все пользователи**: и новые, и существующие; при бампе версии политики —
  повторное согласие.
- Enforcement — **серверный**: без актуального согласия login отвечает признаком
  `policy_required`, а `handshake/init` не выдаёт launch-token.
- Хранение текста — **вариант A**: markdown в репозитории + `go:embed`, версия —
  явная константа. Обновление текста = коммит + деплой (совместимо с git-деплоем
  «прод = точная копия origin/main»). Без редактора в дашборде.
- Черновик текста политики пишет Claude (не юридическая консультация — перед
  публикацией вычитать).

## 1. Бэкенд — новый пакет `internal/policy`

Файлы: `service.go` (логика) + `handler.go` (роуты) — по образцу остальных доменов.

- **Текст:** `backend/internal/policy/privacy.md`, встраивается `go:embed`.
  Версия — `const Version = 1` рядом с embed. Содержательная правка текста →
  бамп константы → после деплоя все проходят согласие заново. Косметическая
  правка (опечатки) — без бампа.
- **HTML-страница:** markdown рендерится в HTML один раз при старте
  (`github.com/yuin/goldmark`), оборачивается в минимальный шаблон.

### Модель данных

Поля в `models.User` (быстрая проверка без join):

```go
PolicyAcceptedVersion int        `gorm:"not null;default:0" json:"policyAcceptedVersion"`
PolicyAcceptedAt      *time.Time `json:"policyAcceptedAt,omitempty"`
```

Новая таблица-журнал `models.PolicyConsent` (юридический след, только append):

```go
type PolicyConsent struct {
    ID         uint      `gorm:"primaryKey"`
    UserID     string    `gorm:"type:uuid;index;not null"`
    Version    int       `gorm:"not null"`
    AcceptedAt time.Time `gorm:"not null"`
    Source     string    `gorm:"size:16;not null"` // launcher | bot
    IP         string    `gorm:"size:64"`
}
```

Схема — через существующий GORM AutoMigrate в `cmd/server/main.go` (и `cmd/bot`,
если бот мигрирует сам).

### Роуты

- `GET /api/policy` (публичный) → `{version, text, updatedAt}` — лаунчер тянет текст.
- `POST /api/policy/accept` (JWT, `auth.RequireAuth`) → тело `{version}`; если
  `version != policy.Version` — 409 (клиент показывает свежий текст заново);
  иначе обновляет поля юзера + пишет строку журнала (`source=launcher`,
  IP из X-Forwarded-For).
- `GET /privacy` (публичный, вне `/api`) → HTML-страница полного текста.

### Ответ логина

`/api/auth/login` (LoginResult) дополняется блоком:

```json
"policy": { "required": true, "version": 1 }
```

`required = user.PolicyAcceptedVersion < policy.Version`.

## 2. Серверный блок запуска игры

В `anticheat` `POST /handshake/init` — сразу после существующей проверки
форс-апдейта (426) — новая проверка:

```
user.PolicyAcceptedVersion < policy.Version → HTTP 451 {"code":"policy_required"}
```

Launch-token не выдаётся → yggdrasil `/join` недостижим → играть без согласия
нельзя, независимо от версии/модификации клиента.

**Гоча (не сломать):** авторизация в anticheat handler — per-route, НЕ `group.Use`.
Проверка политики добавляется внутрь обработчика `init` (или его middleware),
не трогая схему авторизации launch-token-роутов.

## 3. Telegram-бот

- **Гейт в диспетчере:** в `HandleCallback` после `menuViewFor` — если
  `v.User != nil` и `v.User.PolicyAcceptedVersion < policy.Version`, любой колбэк
  (кроме колбэка принятия `cbPolicyAccept`) рисует экран политики вместо
  запрошенного. Тот же гейт — на входе в текстовые флоу (смена пароля, e-mail
  и т.п.) и в командах, открывающих меню.
- **Экран политики:** краткая выжимка (что собираем, включая скриншоты) +
  URL-кнопка «📄 Читать полностью» на `https://launcher.likonchik.xyz/privacy` +
  кнопка «✅ Принимаю». Принял → запись согласия (`source=bot`, IP пустой) →
  главный экран меню.
- **Регистрация:** шаг согласия ПЕРЕД вводом логина; без «Принимаю» флоу не
  стартует. Согласие фиксируется в журнал при создании аккаунта (в тот же
  момент проставляются поля юзера).
- Бот работает с БД напрямую (общий `models` + сервис) — HTTP к серверу не нужен.

## 4. Десктоп-лаунчер (Slint)

Новый экран между логином и списком профилей:

- Если в ответе логина `policy.required` — лаунчер грузит `GET /api/policy`,
  показывает скроллируемый текст, версию/дату, подпись «Продолжая, вы принимаете
  условия политики» и кнопки **«Принять и продолжить»** / **«Выйти»** (разлогин).
- «Принять» → `POST /api/policy/accept {version}` → обычный bootstrap профилей.
  409 (версия успела смениться) → перезагрузить текст и показать снова.
- Ссылка «Открыть в браузере» на `/privacy`.
- **Страховка:** если `handshake/init` вернул 451 `policy_required` (окно раскатки,
  рассинхрон) — лаунчер показывает тот же экран политики, а не сырую ошибку.
- Вёрстка — абсолютным позиционированием с явными `x`/`width` (Slint-гоча:
  layout как один из нескольких детей не фиксирует ширину).

## 5. Раскатка (порядок важен)

1. Деплой бэкенда: политика v1, enforcement активен сразу.
2. Бамп лаунчера до **0.3.3** (`Cargo.toml` + `cargo update -p launcher-slint`),
   сборка `scripts/prod/build-player-launcher.sh --api-url https://launcher.likonchik.xyz`,
   заливка через дашборд сразу как **mandatory**.
3. Старые 0.3.2 при запуске игры получают привычное 426 «обновите лаунчер» →
   автообновление → 0.3.3 показывает экран политики. Окно, когда 0.3.2 получил бы
   «сырое» 451, ограничено минутами между шагами 1 и 2.

## 6. Черновик текста политики (privacy.md)

Разделы:

1. Кто оператор и как связаться (Telegram-контакт администрации).
2. Какие данные собираем: логин, e-mail (если указан), пароль (только bcrypt-хеш),
   Telegram ID/username, HWID (хеш компонентов железа), IP-адрес, история входов,
   игровые сессии, телеметрия и детекты античита, **скриншоты экрана во время
   игровой сессии** (снимаются по запросу администрации для проверки честности игры).
3. Зачем: работа аккаунта, защита от читов и обхода банов, безопасность сервиса.
4. Где и сколько хранится: сервер проекта, сроки (скриншоты — ограниченный срок,
   журнал согласий — весь срок жизни аккаунта).
5. Кому передаём: никому, кроме случаев, предусмотренных законом.
6. Права игрока: запрос копии, удаление аккаунта и данных — через администрацию.
7. Изменения политики: новая версия → повторное согласие в лаунчере/боте.

Пометка: текст готовит ИИ, юридическую вычитку перед публикацией делает владелец.

## Тестирование

- Go (`go vet ./... && go test ./...` через Docker `golang:1.26-bookworm`):
  - `policy`: accept happy-path, повторное принятие, 409 на устаревшую версию,
    запись журнала;
  - `anticheat`: `handshake/init` → 451 без согласия, 200 с согласием
    (не сломать существующие тесты 426);
  - `auth`: блок `policy` в ответе логина;
  - `bot`: гейт колбэков без согласия, экран принятия, шаг согласия в регистрации
    (по образцу `menu_test.go`/`callbacks_test.go`).
- Дашборд не меняется.
- Лаунчер — ручная проверка: логин → экран политики → «Принять» → профили →
  запуск игры; «Выйти» → разлогин; повторный вход экран не показывает.
